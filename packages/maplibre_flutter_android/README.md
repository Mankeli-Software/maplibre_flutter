# maplibre_flutter_android

> ⚠️ **Work in progress — pre-release, not production-ready.** Every target platform
> (Android, iOS, macOS, Windows, Linux, Web) renders a real MapLibre map, but only a small
> slice of the API is wired so far — map creation, camera (get / move / jump / fly), style
> switching, gestures, resize, and lifecycle. Layers, sources, annotations, events, and
> queries are **not** exposed yet. The public API will change without notice; pin an exact
> version. ⭐ the [repository](https://github.com/Mankeli-Software/maplibre_flutter) to follow
> along.

The Android implementation of [`maplibre_flutter`](../maplibre_flutter). Don't depend on this
package directly — depend on `maplibre_flutter`, which endorses it for Android.

## How it works

Android is part of the **mobile tier**: it wraps the official, mature
[MapLibre Android SDK](https://github.com/maplibre/maplibre-native) (`org.maplibre.gl:android-sdk`,
**11.11.0**) rather than the desktop C++ engine. The SDK owns rendering, gestures, inertia, and
lifecycle — the native feel users expect.

### Rendering — `AndroidView` via Hybrid Composition

The native plugin registers a `PlatformViewFactory` that builds the Android SDK's `MapView`
(`org.maplibre.android.maps.MapView`) from the `creationParams` (initial style + camera) handed through the platform-view
registrar. The app-facing widget embeds it with **Hybrid Composition** (`PlatformViewLink` +
`PlatformViewsService.initSurfaceAndroidView`), so the map's `SurfaceView` composites directly in
the view hierarchy with correct gestures and accessibility. On capable devices (Flutter 3.44+,
Android API 34+, Vulkan) this is transparently upgraded to **Hybrid Composition Plus (HCPP)** when
the app sets `io.flutter.embedding.android.EnableHcpp`; otherwise it falls back to classic Hybrid
Composition.

### Control — jnigen, no method channel

Camera and style flow **Dart → JNI → a thin Java shim → the SDK**, with no method channel on the
data path. Dart mints a `mapId`, passes it via `creationParams`; the native view registers its
controller under that id; the Dart controller looks it up over **jnigen** bindings and calls
`getCamera` / `moveCamera` / `setStyle`. The MapLibre Android SDK is **main-thread only**, so the
controller marshals every command onto the main looper and serves cheap camera reads from a state
cache updated by an idle listener.

### Why the bound shim classes are Java, not Kotlin

The two jnigen-summarised classes (`MapRegistry`, `MapLibreController`) live under
`android/src/main/java/` as **Java**, even though the rest of the module is Kotlin. jnigen 0.16's
summarizer can't read the Kotlin 2.3 metadata that AGP 9's built-in Kotlin emits, so the classes it
must read are kept in Java; only the listed classes are summarised, and referenced `org.maplibre.*`
types stay opaque. The bound surface is intentionally kept to primitives + `String` so SDK types are
never summarised. Bindings are committed (`lib/src/maplibre_flutter_android_bindings.g.dart`,
generated by `tool/jnigen.dart`) and regenerated after the Android project is built.

## Toolchain

Built for **AGP 9+**: a `namespace` in Gradle, **built-in Kotlin** (the `kotlin-android` plugin is
deliberately *not* applied — AGP 9 fails the build if it is), Java 17, and consumer R8/ProGuard keep
rules for the jnigen-bound classes (which are `@Keep`-annotated). It stays back-compatible with the
older AGP versions current Flutter still supports.
