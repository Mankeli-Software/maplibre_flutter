# Contributing to maplibre_flutter

Thanks for your interest. This is an early-stage project — the architecture is in
place and every platform renders, but the API surface is small and moving fast. Issues,
discussions, and PRs are welcome.

This document covers local development: repository layout, building, code generation, and
the (not-yet-automated) release flow. For the project overview and current status, see the
[README](README.md).

---

## Repository layout

`maplibre_flutter` is a [melos](https://melos.invertase.dev)-managed monorepo and a Dart
**pub workspace**. App users depend only on `maplibre_flutter`; everything else is an
implementation detail behind the platform interface.

```
maplibre_flutter/                          # repo root: pub workspace + melos config
└─ packages/
   ├─ maplibre_flutter/                     # app-facing API + MapLibreMap widget; endorses impls
   │  └─ example/                           # shared example app + manual test harness
   ├─ maplibre_flutter_platform_interface/  # render-agnostic contract
   ├─ maplibre_flutter_core/                # C ABI shim over mbgl-core + ffigen bindings (desktop)
   ├─ maplibre_flutter_android/             # jnigen → MapLibre Android SDK
   ├─ maplibre_flutter_ios/                 # swiftgen → MapLibre Apple SDK
   ├─ maplibre_flutter_macos/               # ffigen → mbgl-core + Metal texture
   ├─ maplibre_flutter_windows/             # ffigen → mbgl-core + Vulkan texture
   ├─ maplibre_flutter_linux/               # ffigen → mbgl-core + OpenGL texture
   └─ maplibre_flutter_web/                 # maplibre-gl-js via JS interop
```

`mbgl-core` (MapLibre Native) is vendored as a git **submodule** under
`maplibre_flutter_core/third_party/` and built from source via CMake.

## Prerequisites

- Flutter `>=3.41.0` (stable), Dart `>=3.12.2`.
- Per platform: the usual native toolchains — Android Studio / NDK, Xcode (iOS/macOS),
  Visual Studio 2022 + "Desktop development with C++" (Windows), GTK/clang/CMake (Linux),
  Chrome (web).
- For the desktop core (`maplibre_flutter_core`): a C++ toolchain, CMake, and the vendored
  submodule. On Windows, [vcpkg](https://vcpkg.io) — the build hook installs the native
  dependencies into it.

Vendor the native engine once before building any desktop platform from this repo:

```bash
git submodule update --init --recursive
```

## Workspace setup

```bash
flutter pub get        # resolves all workspace members at once (use flutter, not dart)
```

melos is a dev-dependency, so its scripts run via `dart run melos`:

```bash
dart run melos run analyze                 # dart analyze --fatal-infos across all packages
dart run melos run format                  # dart format
dart run melos run test --no-select        # Flutter package tests (non-interactive)
dart run melos run test:native             # pure-Dart/FFI tests (maplibre_flutter_core)
dart run melos run test:web                # browser tests (flutter test --platform chrome)
```

> melos is pinned to `7.8.1` (the newest release still on `cli_util ^0.4.2`, which `ffigen`
> requires; a pub workspace shares a single resolution). melos config lives in the root
> `pubspec.yaml` `melos:` block, not a `melos.yaml`.

### Running the example

The example app is the primary manual test harness. It shows a map with zoom, fly-to, and
style-toggle controls.

```bash
cd packages/maplibre_flutter/example
flutter run -d <android|ios|macos|windows|linux|chrome>
```

## Code generation

Bindings are generated by a Dart script in each package's `tool/` directory and **committed**
to the repo. Never hand-edit the generated `*.g.dart` files — regenerate instead.

```bash
# from the owning package
dart run tool/jnigen.dart      # Android — build the example's Android project once first
dart run tool/swiftgen.dart    # iOS
dart run tool/ffigen.dart      # desktop core (needs libclang)
```

- **jnigen** reads the Android module's compiled release jar, so the Android project must be
  built at least once (and rebuilt after changing a bound class) before regenerating.
- **swiftgen** is experimental; pin versions and expect rough edges.
- **ffigen** needs libclang available locally.

Each per-package README documents the binding-generator quirks specific to that platform.

## Apple: Swift Package Manager and CocoaPods

The Apple packages support **both** SPM and CocoaPods, so neither migrated nor unmigrated
apps break. Toggle SPM for testing:

```bash
flutter config --enable-swift-package-manager     # then build the example
flutter config --no-enable-swift-package-manager  # CocoaPods path
```

Flutter applies the SPM/Pod integration at build time (ephemeral). The resulting churn in the
example's `ios`/`macos` project files is regenerated on demand and **should not be committed**;
`git checkout` it after a build.

### iOS & macOS codesigning (running on a physical device)

To avoid committing personal Apple Development Team IDs, the example injects signing config from
a local, git-ignored file:

1. Create `packages/maplibre_flutter/example/ios/Flutter/Local.xcconfig` (already git-ignored).
2. Add your Team ID:
   ```properties
   DEVELOPMENT_TEAM = YOUR_TEAM_ID
   ```

`Debug.xcconfig` / `Release.xcconfig` pull it in via `#include? "Local.xcconfig"`. The `?` makes
it optional, so CI and the simulator build without the file. The Xcode project deliberately
carries no `DEVELOPMENT_TEAM`.

### Editing the iOS Swift sources in VS Code

The plugin's iOS Swift sources depend on the Flutter SDK package, which Flutter only
materialises during a build. In **Xcode** just open the example's `Runner.xcworkspace`. In
**VS Code**, the Swift LSP reports `No such module 'MapLibre'` until the package resolves
standalone — one-time setup after building the example iOS app at least once:

```bash
cd packages/maplibre_flutter_ios/ios
ln -sfn ../../maplibre_flutter/example/ios/Flutter/ephemeral/Packages/.packages/FlutterFramework FlutterFramework
(cd maplibre_flutter_ios && swift package resolve)
```

Reload the VS Code window. The symlink and `Package.resolved` are git-ignored; the symlink only
dangles after `flutter clean` — rebuild to restore it.

## Coding conventions

- **Brand casing.** Dart identifiers use `MapLibre` (capital L): `MapLibreMap`,
  `MapLibreMapController`. Package names, file names, asset ids, and pubspec keys stay
  snake_case (`maplibre_flutter`).
- `dart format` + `flutter analyze` must be clean.
- [Conventional Commits](https://www.conventionalcommits.org) — they drive `melos version`.
- No method channels on the data path; bindings only. Platform-channel use is limited to what
  the texture/platform-view registrar requires.
- Always pin to the newest stable of fast-moving tools (jni, ffigen, swiftgen, melos, the
  MapLibre SDKs). Check pub.dev before adding or bumping a dependency.

## Testing

Tests are a first-class goal of this project. The layers:

1. **Platform-interface unit tests** — the contract, exercised with a mock implementation.
2. **Dart-wrapper unit tests** — each platform wrapper with its generated bindings mocked,
   verifying the right native calls and callback/lifecycle reference management.
3. **Widget / golden tests** — the `MapLibreMap` widget picks the correct embed (platform view
   vs `Texture` vs `HtmlElementView`) and disposes cleanly.
4. **Native unit tests** — Kotlin (JUnit), Swift (XCTest), C++ shim (ctest/GoogleTest).
5. **Integration tests** (`integration_test/`) — a real map on a real device/engine: style
   loads, camera moves, a frame is actually produced.

When you add a feature, add the test that proves it. The
[feature matrix](FEATURE_MATRIX.md) only marks a cell supported when a verified example run or
an automated test backs it.

## Releasing

> CI is not yet enabled (the workflow files exist but their automatic triggers are commented
> out). Releases are currently manual.

All nine federated packages are publishable; only the workspace root and `example` stay
private. Siblings depend on each other by **version constraint** (`^x.y.z`), not `path:` — the
pub workspace links them locally for dev, and melos keeps the constraints in lockstep on bump.
Publish in **dependency order** (platform interface → core → implementations → app-facing).

```bash
melos version                          # coordinated bump from Conventional Commits
dart run melos run publish:dry-run     # dart pub publish --dry-run everywhere — must be clean
melos publish                          # publish in dependency order
```
